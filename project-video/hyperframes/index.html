Can users trust an AI game they cannot see?

SocialCOMPACT is not just a result page. It is a multi-Agent Survivor match where conversation, prediction, and action need to become visible evidence.

In this 5-minute version comparison, we show how the codebase moved from "it runs" to "users can inspect how it runs."
Before the architecture details, here is the problem: if a multi-Agent match is invisible, users only wait. The redesign turns chat, prediction, decisions, and results into visible evidence.
Project overview

What the project does, who uses it, and what the UI returns.

Problem

SocialCOMPACT evaluates social intelligence in AI agents by running a Survivor-style multi-agent game.

Inputs
  • player names
  • round count
  • backend URL
  • Agent service URLs
Outputs
  • match result
  • player state cards
  • live logs
  • Agent artifacts
The user enters match settings in the browser, starts a Survivor game, and expects more than a final winner: they want visible logs, player state, and a readable explanation of what the Agents are doing.
Overall architecture

One match travels through browser request, backend coordination, Arena execution, and frontend polling.

Browser submits settings -> backend creates match_id -> Arena runs in background -> frontend polls UI-ready state and logs.
At a high level, the browser only sends configuration. The backend owns match creation and orchestration. Arena and Agents generate raw artifacts. The frontend keeps polling until the run becomes a readable result page.
Version 1 explanation

Version 1 proved the web flow could run, but the long match stayed mostly opaque.

What V1 already achieved
  • start a web match
  • wait for the game to finish
  • show a final result page
queued / running / final result

The user could tell that a run existed, but not what the Agents were thinking or doing.

Main limitations
  • little observability during the run
  • public access still fragile
  • localhost dependency hurt real usability
Version 1 was an important first milestone. It made the web flow possible, but it still behaved like a black box: users mostly waited for a final answer instead of seeing the match process unfold.
Evidence from real blockers

Version 1 exposed four system problems, not just isolated tooling errors.

Version traceability under team pressure
We needed a stable branch-and-push flow so evidence, commits, and deployment updates stayed auditable.
Frontend build reproducibility
A web demo is not trustworthy if teammates cannot rebuild the same frontend consistently.
One long request could not drive a web match
The browser needed match_id plus polling instead of waiting for a blocking Arena run to finish.
Localhost was not a public multi-service product
We had to move from a local experiment to public Vercel and Render services with health checks and fallback.
The visible failures were only the surface. Underneath them were four deeper engineering problems: traceable collaboration, reproducible frontend builds, asynchronous web orchestration, and public multi-service deployment. Those are the reasons Version 2 had to be redesigned.
Version 2 explanation

The backend became the real coordinator: fast reply, background run, state storage, polling APIs.

input: match settings processing: orchestrate Arena in background output: UI-ready JSON for results page
Version 2 is not just a prettier frontend. The deeper change is in the backend: it creates a match id immediately, launches Arena asynchronously, stores structured events, and serves stable polling APIs to the UI.
A2A communication path

Arena brokers the Agent conversation instead of letting the two Agents call each other directly.

Backend -> Arena -> Agent endpoint -> streamed artifacts -> normalized logs -> frontend polling
This is where the A2A logic matters. Arena is the conversation hub. It preserves context per Agent URL, sends prompts to each Agent endpoint, collects replies, and streams artifacts back upstream to the backend.
Game logic

The Survivor loop is not one action. Each round moves through chat, prediction, decision, action, and observation.

talk predict + decide process actions + send observations
The game itself also becomes more understandable in Version 2. Instead of treating the match as one mysterious result, the video can now show the round loop: communication, prediction, action processing, and feedback.

Reflection and group process

Version 2 is more explainable and more usable, but it costs more operational complexity.

Next steps: add persistent replay storage, stronger tie handling, and a preflight health check before each match. The final split also became clearer: deployment, backend orchestration, UI, logs, and mini theater work were owned by different teammates.

A: GitHub workflow, homepage, Render deployment B: start page, match form, mini theater character design C: results overview, player cards, mini theater layout D: live logs, timeline, mini theater animation E: FastAPI backend, Arena integration, web orchestration F: Vercel deployment, public verification, final video
The main trade-off is operational complexity: more services, more health concerns, and more cold-start risk. But that trade-off bought us a public demo users can actually understand. Our group process mattered too: A handled coordination and Render deployment, B/C/D split the mini theater and results experience, E owned backend orchestration, and F handled Vercel deployment plus the final video.
